# План тестирования — «симулятор первым»

Главный принцип: **на реальные часы и телефон ничего не ставится, пока не пройдены все
симуляционные сценарии ниже.** Хардверный проход (раздел 4) — финальная стадия и сведён к
тому, что симулятор физически не может воспроизвести.

Последовательность гейтов:
**Gate 1** (юнит-тесты Android) → **Gate 2** (Connect IQ Simulator, watch-логика) →
**Gate 3** (сквозной протокол через adb-tethered) → **Gate 4** (железо).
Переход к следующему гейту — только при зелёном предыдущем.

---

## 0. Фикстуры (общие тестовые данные)

Сложить в `docs/fixtures/` несколько реальных GPX из Organic Maps + синтетику:

| Файл | Что проверяет |
|---|---|
| `short_city.gpx` | короткий городской маршрут, много поворотов |
| `long_trail.gpx` | длинный (десятки км) — стресс по числу точек/чанков |
| `straight.gpx` | почти прямой — манёвров почти нет (граница `TURN_MIN`) |
| `hairpin.gpx` | серпантин — резкие повороты, близко стоящие манёвры (тест `MERGE_DIST`) |
| `single_point.gpx` | вырожденный: одна точка |
| `empty.gpx` | пустой/битый файл |
| `geojson_line.geojson` | тот же маршрут в GeoJSON |
| `loop.gpx` | замкнутый маршрут (старт = финиш), проверка off-route у самопересечения |

## 1. Gate 1 — Android, юнит-тесты (JVM, без эмулятора)

Чистая логика, гоняется в `./gradlew test`. Эмулятор не нужен.

**Парсеры**
- GPX `<trkpt>` → корректный список точек; `<rtept>` как fallback.
- GeoJSON `LineString` → корректный список.
- Битый/пустой файл → понятная ошибка, не краш.

**Децимация (Douglas–Peucker)**
- Число точек ≤ `MAX_LINE_POINTS`.
- Форма сохранена: максимальное отклонение децимированной линии от исходной < заданного порога.
- Вырожденные входы (1–2 точки) не падают.

**Детект манёвров**
- На `straight.gpx` манёвров ≈ 0 (только финиш).
- На `hairpin.gpx` повороты найдены, правильный знак (лево/право) и тип (плавный/обычный/резкий).
- `MERGE_DIST`: близкие повороты схлопываются.
- `dist` монотонно растёт; последний манёвр — `type=9`.
- `idx` манёвра указывает на валидную точку децимированной линии.

**Кодирование**
- deg→микроградусы→deg round-trip в пределах допустимой ошибки.
- Дельты восстанавливаются накопительной суммой обратно в абсолютные.
- Сериализованный размер чанка при `POINTS_PER_CHUNK` ниже целевого порога.

## 2. Gate 2 — Connect IQ Simulator, watch-логика

Запуск watch-аппа в симуляторе SDK. Никакого телефона ещё нет — сообщения подаём моками.

**Сборка маршрута**
- Скормить заранее заготовленную последовательность `H → L0..Ln → M → E` (фиксированные
  Dictionary) и проверить: маршрут собран полностью, `np`/`nm` сходятся, состояние `READY`.
- Пропустить один `L` → апп шлёт `N(seq)` и корректно дособирает после повтора.
- Перепутать порядок чанков → корректная укладка по `s`.

**Рендеринг и следование (через симуляцию GPS)**
- Включить в симуляторе проигрывание трека (Simulate Data → позиция движется по маршруту).
- Линия рисуется, «я» в центре, heading-up поворачивается корректно.
- Баннер «следующий поворот N м» отсчитывает дистанцию вниз и переключается на следующий
  манёвр после прохождения.
- Вибро у поворота срабатывает один раз на манёвр.
- Сместить симулируемую позицию в сторону от маршрута → через `OFFROUTE_T` появляется
  «OFF ROUTE» + одиночное вибро; возврат на маршрут — баннер гаснет.
- Дойти до финиша → «Прибытие».
- `loop.gpx`: у самопересечения «следующий поворот» не перескакивает на дальнюю ветку.

**Граничные**
- Маршрут из 1 точки / пустой `M` → апп не падает, осмысленное сообщение.
- Потеря GPS (accuracy = UsableAccuracy.NOT_AVAILABLE в симуляции) → апп не крашится,
  показывает «нет GPS», возобновляет при возврате сигнала.

## 3. Gate 3 — сквозной протокол через adb-tethered

Реальный Android-апп ↔ **симулятор** (стоит вместо часов). Физических часов ещё нет.

Настройка:
- В Android debug-сборке тип подключения `TETHERED` (`BuildConfig.CIQ_CONNECT_TYPE`).
- `adb forward tcp:7381 tcp:7381`.
- Запустить watch-апп в симуляторе, затем Android-апп (на USB-телефоне или эмуляторе).
- Симулятор должен появиться в списке устройств Android-аппа.

Сценарии:
- Шаринг `short_city.gpx` в апп → полная передача `H/L/M/E` с ack → маршрут виден в симуляторе.
- `long_trail.gpx` → много чанков, прогресс-бар идёт до конца, маршрут собран.
- Симуляция потери ack: временно «проглотить» ответ → телефон ретраит по таймауту, затем успех.
- Исчерпание ретраев → телефон показывает ошибку + Retry, повторная отправка с `H` проходит.
- Несовпадение версии протокола `v` → симулятор показывает «Update app», передача игнорируется.

> Что Gate 3 НЕ проверяет: реальный лимит размера сообщения и реальную надёжность BLE —
> симулятор тут «добрее» железа. Это уходит в Gate 4.

## 4. Gate 4 — железо (финальная стадия, только после зелёных 1–3)

Минимальный проход на реальном Instinct 3 + телефоне с Garmin Connect Mobile.

Чек-лист:
- [ ] Собрать `.prg` под реальный device id Instinct 3 (из `<sdk>/bin/devices.xml`),
      скопировать в `\GARMIN\APPS\`, открыть виджет.
- [ ] Android release-сборка с `WIRELESS`, часы спарены, GCM запущен.
- [ ] **Подбор `POINTS_PER_CHUNK`:** начать с консервативного значения, увеличивать до отказа,
      зафиксировать безопасный максимум. Обновить конфиг.
- [ ] **Тайминги BLE:** проверить реальные задержки, поведение при `FAILURE_UNKNOWN`,
      докрутить `ACK_TIMEOUT`/`MAX_RETRY`.
- [ ] Передать `short_city.gpx` и `long_trail.gpx` целиком, без потерь.
- [ ] **GPS heading** на скорости ходьбы: есть ли курс из Position или нужен компас-fallback.
- [ ] Читаемость линии и баннера на 50-мм экране на улице (солнце).
- [ ] Расход батареи за ~1 час навигации (ориентир — сопоставимо с breadcrumb-аппами).
- [ ] Пройти короткий реальный маршрут целиком: повороты, вибро, off-route, прибытие.

Все подобранные на железе значения вернуть в конфиг и снять метки `// TODO: verify on device`.

## Definition of Done (MVP)

Gate 1–3 зелёные **и** Gate 4 чек-лист пройден на реальном маршруте: построил в Organic →
расшарил → открыл виджет → дошёл по линии с подсказками без сбоев передачи.
