# my_routecast — техническое задание

Передача маршрута с Android на часы **Garmin Instinct 3 Tactical Solar 50 mm** и отрисовка
его на часах: линия маршрута (heading-up) + псевдо-turn-by-turn (стрелка следующего поворота +
дистанция + вибро) + предупреждение о сходе с маршрута. Всё оффлайн, без Connect IQ Store,
без нативной навигации Garmin (рисуем сами).

---

## 1. Зафиксированные решения

| Тема | Решение |
|---|---|
| Источник маршрута | **Только Organic Maps.** Маршрут строится в Organic, сохраняется как трек, шарится файлом. Google Maps исключён. |
| Форматы входа | GPX (основной), GeoJSON (легко). KMZ (zip+KML) — за флагом, не в MVP. |
| Транспорт телефон→часы | Connect IQ Mobile SDK для Android (`com.garmin.connectiq:ciq-companion-app-sdk`). Работает поверх Garmin Connect Mobile (GCM как BLE-мост). |
| Запуск на часах | Тип приложения — **виджет**. Пользователь сам открывает виджет ПЕРЕД шарингом. Мгновенного «пробуждения с телефона» не делаем (его в CIQ нет). |
| Навигация | Линия **И** псевдо-TBT. TBT считается из геометрии полилинии (без названий улиц). |
| Установка | Сайдлоад: `.prg` → `\GARMIN\APPS\` по USB. APK компаньона — через adb. |
| Карта-подложка | Нет (Instinct без базовой карты). Рисуем только линию, точку «я», стрелку, баннер поворота. |
| Порядок проверки | **Сначала симулятор, потом железо.** Установка на реальные часы/телефон — только после прохождения всех симуляционных сценариев. Это стейт-гейт, а не пожелание. |

> **Принцип «симулятор первым».** Перед любой установкой на физические устройства проект
> обязан пройти полный набор симуляций из `docs/testing.md` (юнит-тесты Android, сценарии
> Connect IQ Simulator, сквозной прогон протокола через adb-tethered). Хардверный проход —
> финальная стадия, и она ограничена только тем, что симулятор физически не воспроизводит
> (см. §7). Реализацию вести так, чтобы каждую фичу можно было проверить в симуляторе до
> железа.

## 2. Поток данных

```
Organic Maps  --(построил маршрут → сохранил как трек → Share GPX)-->
Android companion (my_routecast)
    1. ловит файл через ACTION_SEND / ACTION_VIEW (intent-filter)
    2. парсит в список точек [lat, lon]
    3. децимирует (Douglas–Peucker) до ~150–300 точек для линии
    4. детектит манёвры (изменение курса) → компактный список поворотов
    5. шлёт на часы чанками через ConnectIQ.sendMessage (stop-and-wait + ack)
        --(BLE через GCM)-->
Watch widget (Monkey C)
    6. mailbox-listener собирает чанки → восстанавливает маршрут + список поворотов
    7. Position API даёт текущие координаты/курс
    8. рендер: линия heading-up, «я» в центре, баннер «следующий поворот N м»
    9. off-route alert + вибро у поворота
```

Ключевое ограничение, под которое спроектирован протокол: **сообщения CIQ маленькие**
(порядок единиц КБ; фоновые данные ~8 КБ, wake-сообщение 255 Б). Поэтому маршрут НЕ влезает
в одно сообщение — обязательны децимация + чанкинг.

## 3. Протокол сообщений (phone ↔ watch)

Каждое сообщение — Monkey C `Dictionary`. Поле `t` — тип. Все координаты — целые
**микроградусы** (`round(deg * 1e6)`), чтобы не гонять float и экономить размер.

| `t` | Направление | Поля | Назначение |
|---|---|---|---|
| `"H"` | phone→watch | `v` (версия протокола), `np` (кол-во точек линии), `nc` (кол-во чанков), `nm` (кол-во манёвров), `name` | Хендшейк/начало передачи. Часы сбрасывают буфер. |
| `"L"` | phone→watch | `s` (seq чанка, 0-based), `p` (массив `[dlat, dlon]` — дельты в микроградусах от предыдущей точки) | Чанк линии. ~40 точек на чанк. Дельты вместо абсолютных значений = меньше байт. |
| `"M"` | phone→watch | `m` (массив манёвров, см. ниже) | Список поворотов. Обычно влезает в 1 сообщение. |
| `"E"` | phone→watch | — | Конец передачи. Часы валидируют, что собрано `np` точек и `nm` манёвров. |
| `"A"` | watch→phone | `s` (seq, который успешно принят; для `H`=-1, для `M`=-2, для `E`=-3) | Ack. Телефон шлёт следующий пакет только после ack. |
| `"N"` | watch→phone | `s` | Nack/таймаут — переслать пакет `s`. |

Манёвр (элемент массива `m`):
```
[ idx,   // индекс точки линии, где поворот
  type,  // 0=прямо/слияние, 1=плавно вправо, 2=вправо, 3=резко вправо,
         //                   -1=плавно влево, -2=влево, -3=резко влево, 9=финиш
  dist ] // накопленная дистанция от старта до манёвра, в метрах (целое)
```

Надёжность: **stop-and-wait**. Телефон отправляет `H`, ждёт ack; затем по очереди `L0..Ln`
(каждый ждёт ack), затем `M`, затем `E`. Таймаут пакета — 4 с, до 3 ретраев, потом — ошибка
в UI телефона. Это переживает обычные дропы BLE (в форумах частый `FAILURE_UNKNOWN`).

Подробная схема с примерами байтовых размеров — в `docs/protocol.md`.

## 4. Android-компаньон (`android/`)

**Язык/стек:** Kotlin, minSdk 26+, один Activity + фоновый sender. Зависимость:
`com.garmin.connectiq:ciq-companion-app-sdk` (Maven Central).

### 4.1 Приём файла
`intent-filter` на:
- `ACTION_SEND` / `ACTION_VIEW`, mime `application/gpx+xml`, `application/geo+json`,
  `application/json`, `application/vnd.google-earth.kmz` (последний — за флагом `ENABLE_KMZ`).
- Плюс fallback по расширению `.gpx`, `.geojson`, `.json`.

Так апп появляется в системном «Поделиться» и в «Открыть с помощью» рядом с Organic Maps.

### 4.2 Парсеры
- **GPX:** читаем `<trkpt lat lon>` (и `<rtept>` как fallback). Игнорируем высоту/время.
- **GeoJSON:** `LineString` / первый `Feature` с геометрией линии.
- **KMZ (флаг):** unzip → `doc.kml` → `<coordinates>`.

Результат единый: `List<Pt(lat: Double, lon: Double)>`.

### 4.3 Обработка
1. **Децимация** Douglas–Peucker, epsilon подбирается так, чтобы точек стало
   ≤ `MAX_LINE_POINTS` (по умолчанию 250). Сохраняем форму, режем плотность.
2. **Детект манёвров — нетто-кластеризация** по *неразреженной* линии (линию на карте НЕ
   упрощаем — реальные изгибы важны, напр. тропинки в лесу). Считаем не каждую вершину
   отдельно, а поворотные **события**:
   - bearing каждого сегмента; в вершине `i`: `turn = normalize(bearing[i]-bearing[i-1])`;
   - вершина с `|turn| ≥ MIN_VERTEX (12°)` входит в текущее событие; **знаковые углы события
     суммируются** (`acc`); событие закрывается прямым участком `≥ MERGE_DIST (25 м)`;
   - манёвр выдаём, только если `|acc| ≥ TURN_MIN (35°)` — иначе это виляние (переход через
     дорогу: лево+право≈0) и оно гасится. Так на реальном треке число подсказок резко падает;
   - угол манёвра = `acc`; классификация по модулю: 35–70 плавный, 70–120 обычный, >120 резкий;
     знак → лево/право; `idx` — пик события, привязан к ближайшей точке *децимированной* линии;
   - последняя точка маршрута → манёвр `type=9` (финиш); `dist` = накопленная длина до пика.
3. **Кодирование** в микроградусы и дельты (см. протокол).

### 4.4 Передача
- Получить список устройств через SDK, выбрать Instinct 3 (если одно — авто).
- Слать по протоколу stop-and-wait. UUID watch-аппа берётся из манифеста watch-проекта
  (одинаковый на обеих сторонах) — вынести в `BuildConfig`/`Constants`.
- **Режим подключения — билд-флаг.** ConnectIQ-инстанс создаётся с типом:
  `TETHERED` (через adb к симулятору) для debug-сборки и `WIRELESS` (BLE через GCM) для
  release. Дефолт SDK — `WIRELESS`, поэтому для теста на ПК это переключается явно.
  Реализовать через `BuildConfig.CIQ_CONNECT_TYPE`, чтобы тот же код тестировался в
  симуляторе и работал на железе без правок.
- UI минимальный: имя маршрута, кол-во точек, статус устройства, прогресс-бар чанков,
  кнопка Retry. Никаких карт.

### 4.5 Гард по размеру
Перед отправкой посчитать примерный размер сериализованного чанка; если близко к лимиту —
уменьшить `POINTS_PER_CHUNK`. Значения вынести в конфиг и **выверить на железе** (см. §7).

## 5. Watch-виджет (`watch/`)

**Тип:** widget (быстро открывается с циферблата). Monkey C, target — Instinct 3.
Точный device id — найти в `<sdk>/bin/devices.xml` (НЕ выдумывать строку).

> **Экран — монохромный MIP.** Instinct 3 Tactical Solar 50 mm имеет монохромный
> sunlight-visible transflective MIP-дисплей (НЕ цветной AMOLED). Точные параметры из CIQ SDK
> (профиль `instinct3solar45mm`, общий для 45/50 mm): **176×176 px, форма `semi-octagon`,
> без тача** (управление только кнопками). Отсюда два жёстких следствия для рендера:
> (1) различать сущности **нельзя цветом** — только формой, толщиной линии, пунктиром и
> маркерами; (2) текст и стрелка
> поворота должны быть **крупными** (баннер ≈ треть экрана), мелкие подписи не читаются на
> улице. Точный размер/разрешение экрана брать из `devices.xml`, не хардкодить.

### 5.1 Состояния
`IDLE` (ждём передачу) → `RECEIVING` (собираем чанки, показываем прогресс) →
`READY` (маршрут собран — показываем **обзор всего маршрута** fit-to-screen со старт/финиш
маркерами; приглашение «Старт» — крупной стрелкой `▶` в субэкране, без дублирующей подписи)
→ `NAVIGATING` (отрисовка по позиции) → `ARRIVED`
(экран «Прибытие», выход по BACK). Ошибка сборки → `IDLE` с сообщением.

**Управление кнопками (как в нативной навигации Instinct):**
- **START (кнопка GPS)** — старт навигации из `READY`; стоп/пауза из `NAVIGATING` (→ `READY`).
- **UP / DOWN** — зум карты, **3 осмысленных уровня** (индикатор-пипсы + подпись охвата):
  ближний ≈60 м (разглядеть частые/сложные повороты), **средний ≈150 м (по умолчанию)** —
  обычный пешеходный обзор, дальний ≈320 м (понять, что впереди). Конкретные метры — tuning.
- **BACK** — открыть подменю **«Пауза/Продолжить»** и **«Завершить»**; на `ARRIVED` — сразу
  выход. **Завершить очищает маршрут** из памяти (→ `IDLE`, минимум следов после себя).
  «Пауза» отключает GPS-события, «Продолжить» включает обратно. (Обзор всего маршрута —
  отдельным пунктом/жестом, TODO.)
- **SET** — снять текущее предупреждение о повороте/финише вручную (страховка, если зависло).

### 5.2 Приём
- `Communications` mailbox-listener (`registerForPhoneAppMessages` / `Mailbox`).
- На `H` — сбросить буферы, выделить массив на `np` точек.
- На `L` — разложить дельты в абсолютные микроградусы, сложить в буфер по `s`. Отправить `A(s)`.
- На `M` — сохранить манёвры, `A(-2)`.
- На `E` — проверить полноту; ок → `A(-3)`, перейти в `READY`; не ок → `N(пропущенный s)`.

### 5.3 Позиционирование
**GPS — собственный приёмник часов** (Instinct 3 автономен). Телефон нужен только для
разовой передачи маршрута; в навигации часы ведут офлайн, без телефона.
`Position.enableLocationEvents(...)`; из `Position.Info` берём `position`, `heading`
(курс по GPS), `accuracy`. Если heading недоступен на малой скорости — fallback на компас
(`Sensor`/`Toybox.SensorHistory` heading) либо north-up.

### 5.4 Рендер (на каждый onUpdate / позиционный тик)
- Проекция точек в локальную плоскость относительно «я» (equirectangular-приближение,
  на дистанциях навигации погрешность пренебрежима):
  `x = R*cos(lat0)*Δlon`, `y = R*Δlat`.
- **Heading-up:** повернуть все точки на `-heading`. «Я» помещать **не в центр, а ниже**
  (≈70% высоты основного круга) — так впереди по курсу видно больше маршрута.
  (Настройка north-up — через меню кнопкой, без CIQ-settings.)
- **Один тон, без цвета.** Экран одноцветный (1 «чернило» + фон), полутонов нет. Иерархию
  задавать размером и начертанием, «тусклое» — дизерингом/тонкой линией, **акцент и тревогу —
  миганием/инверсией, не цветом**. Никаких цветовых кодировок в рендере.
- Масштаб `px/m` — авто (чтобы было видно ~следующие 300 м) с ручной коррекцией кнопками UP/DOWN.
- Нарисовать полилинию (`dc.drawLine` по сегментам), точку «я» (круг + стрелка курса).
  Экран монохромный (см. рамку выше) → пройденную/оставшуюся части различать **не цветом, а
  начертанием**: оставшаяся — сплошная толстая линия, пройденная — тонкая/пунктир (опционально
  по `nearestSegment`). Манёвры можно метить маркером-точкой на линии, финиш — кольцом.
- **Нижнее поле — формат `NNN/YY`** (крупно, без подписей и единиц, чтобы экономить пиксели):
  `NNN` = дистанция до следующего поворота — **в метрах ниже 500 м, в км (1 знак) от 500 м**;
  `YY` = всего осталось до финиша в км. Пример: `120/4.2`, `0.8/4.2`.
- **Толщина линии** маршрута — параметр читаемости; оставшаяся линия толстая, пройденная —
  тонкий пунктир. Финальные значения **подобрать на эмуляции/железе** под 50-мм MIP.
- **Мини-экран Instinct → курсовая стрелка.** У Instinct 3 в правом-верхнем углу есть
  отдельная круглая врезка (subscreen). Тип приложения — widget, поэтому мы получаем
  **единый `dc` на весь кадр** и адресуем субэкран по координатам (в отличие от дата-филда,
  где он отдельное поле). В субэкране — **heading-relative указатель на look-ahead точку**:
  берём точку на маршруте в `LOOKAHEAD ≈ 40 м` впереди от `traveledDist`, считаем её пеленг
  относительно текущего курса и рисуем стрелку (Y-образную) — она непрерывно «ведёт» и
  доворачивается перед поворотом. Off-route → мигающее `OFF`, финиш → кольцо.
  **Центр и радиус субэкрана брать из `devices.xml`/layout, не хардкодить** (проверяется в
  симуляторе); между основным кругом и врезкой есть зазор — карту через оба не растягивать.
  > Проверить в симуляторе (Gate 2): получает ли **widget** на Instinct 3 пиксели субэкрана в
  > общий `dc`. Если нет — fallback: указатель/баннер в верхней трети основного круга.
- **Карта всегда видна — без полноэкранного баннера.** Полноэкранное предупреждение убрано:
  карта heading-up остаётся на экране постоянно (важна непрерывная ситуативная картинка).
  Сигнал о повороте — в субэкране (ниже) + одна короткая вибрация.
- **Тип поворота — кратко в субэкране.** При `dist_to_turn ≤ TURN_WARN_M` (≈50 м) субэкран на
  `WARN_HOLD ≈ 2.5 с` показывает **схематичную стрелку типа манёвра** (плавно / ≈90° / резко +
  сторона — из `type`), потом возвращается к пеленгу. Метров на стрелке не пишем (дистанция —
  в нижнем поле). Так пеленг (основное) почти всегда на месте, а тип поворота лишь подсвечивается.
- **Опираемся на идиомы нативной навигации Garmin** (страницы Map/Compass), чтобы UX был
  привычным:
  - **Субэкран = bearing pointer (основной режим)**: стрелка-пеленг на look-ahead точку
    (калька с компас-страницы «Go To»). Без рамки (она физическая), без компас-кольца/насечек.
    Кратко перебивается типом поворота за 50 м (см. выше), потом снова пеленг.
    **Стрелка динамическая, heading-relative**: цель проецируется относительно текущего курса
    (GPS/компас) — повернёшься к повороту, станет «вперёд»; развернёшься — покажет назад.
    Карта центрирована по физическому экрану (равные поля), субэкран — вырез в правом-верхнем
    углу того же полотна. **Подтверждено по `simulator.json`:** субэкран — область того же
    176×176 `dc`, ~62×62 px, центр в координатах отрисовки ≈ `(144, 31)`. Виджету доступен
    (рисуем одним `dc`) — ключевой риск Gate 2 снят.
  - **Поля данных в типографике Garmin**: крупное значение без лишних подписей (нижнее
    поле `NNN/YY`, см. выше).
  - **Алерты в их стиле**: слово-метка манёвра над крупной стрелкой и дистанцией;
    `OFF ROUTE` / `Прибытие` — отдельными экранами.
  - **Экран `READY` — обзор всего маршрута** fit-to-screen со старт/финиш маркерами; общая
    длина — одним числом (`4.4 км`), без длинных подписей.
  - НЕ берём то, чего у нас нет по ТЗ: базовую карту-подложку, компас-кольцо вокруг карты,
    названия улиц/именованные course points (рисуем чистую геометрию).

### 5.5 Логика следования
- Каждый тик: найти ближайшую точку/сегмент маршрута (cross-track), получить
  `traveledDist` и `nextManeuver` (первый с `dist > traveledDist`).
- **Старт с любой точки.** Прогресс определяется проекцией на ближайший сегмент, а не от
  точки 0 — можно начать с середины маршрута, апп подхватит. Если стартовать далеко от линии —
  состояние `OFF ROUTE`, пока не выйдешь на неё. Подвох — самопересечения (петля): ближайший
  сегмент неоднозначен, разрешать предпочтением сегмента по ходу движения (см. `testing.md`).
- **Вибрация — одна короткая на ВСЕ оповещения** (`Attention.vibrate`, ~200 мс). Экономим
  батарею: никаких двойных/паттернов. Программируется (`VibeProfile(интенсивность, мс)`).
- **Off-route:** если расстояние до ближайшего сегмента > `OFFROUTE_M` (по умолч. 40 м)
  дольше `OFFROUTE_T` (5 с) — баннер «OFF ROUTE» + одно короткое вибро. Линия тускнеет,
  **стрелка-пеленг разворачивается к ближайшей точке маршрута** — ведёт назад на линию.
  Нового маршрута не строим (роутинга нет). При возврате в пределах порога — баннер гаснет.
- **Подход к повороту:** при `dist_to_turn ≤ TURN_WARN_M` (≈50 м) — **одно короткое вибро** +
  тип поворота в субэкране на `WARN_HOLD` (см. 5.4). Один сигнал на манёвр; второго на самом
  повороте НЕ даём (карта и так видна — экономим батарею).
- **Финиш:** анонс при `dist_to_finish ≤ TURN_WARN_M` — одно вибро; **прибытие** в пределах
  `ARRIVE_M` (≈20–25 м) — экран «Прибытие» + одно вибро, выход по BACK с **очисткой маршрута**
  (→ `IDLE`).

### 5.6 Батарея
Чистая отрисовка линии дешёвая (сравнимо с dwMap ~2.4%/ч). Не держать максимальную яркость
насильно; перерисовывать по позиционному событию + таймеру 1 Гц, не чаще.

## 6. Объём MVP vs позже

**MVP (цель первой рабочей версии):**
- GPX + GeoJSON вход; децимация; манёвры из геометрии.
- Передача stop-and-wait, прогресс, retry.
- Виджет: приём, линия heading-up, «я» + стрелка, баннер следующего поворота, off-route, вибро.
- Сайдлоад обеих частей.

**Позже / опционально:**
- KMZ; north-up toggle; ручной зум сохраняемый; «развернуть маршрут» (идти в обратную сторону);
  несколько сохранённых маршрутов в памяти виджета; экран обзора всего маршрута (fit-to-screen);
  цветовая дельта высоты.

## 7. Хардверный проход — финальная стадия (gate)

Установка на реальные часы/телефон выполняется **только после** прохождения всех
симуляционных сценариев из `docs/testing.md`. На железо выносится лишь то, что симулятор не
воспроизводит физически:

1. Реальный максимальный размер одного сообщения mailbox на Instinct 3 → подобрать
   `POINTS_PER_CHUNK`.
2. Стабильность связки телефон→GCM→часы, поведение при `FAILURE_UNKNOWN`, нужные таймауты.
3. Точный device id Instinct 3 для сборки `.prg` (из `devices.xml`).
4. Доступность `heading` из Position на ходьбе/беге vs необходимость компаса.
5. Расход батареи и читаемость линии на физическом 50-мм экране.
6. Монтируется ли Instinct 3 как mass storage для копирования `.prg` (ожидаемо да — уже
   используется `GARMIN/NewFiles`).

Полный план тестов (симуляции + этот хардверный чек-лист) — в `docs/testing.md`.

## 8. Сборка и установка (кратко; подробнее в README)

**Watch:** VS Code + Monkey C extension → "Build for Device" → `.prg` → скопировать в
`\GARMIN\APPS\` по USB. Открыть виджет на часах.

**Android:** Android Studio → собрать APK → `adb install`. Дать разрешение на работу с
ConnectIQ/GCM. GCM должен быть установлен и часы спарены.

UUID watch-аппа из `watch/manifest.xml` продублировать в `android` конфиг — стороны должны
совпадать.
